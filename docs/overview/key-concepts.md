# Key Concepts

Every important term used across Vizzly's documentation, defined in one place.

---

**Baseline** — An approved screenshot stored in `.vizzly/baselines/`. It represents "this is what the UI should look like." Every new screenshot is compared against its corresponding baseline. Baselines are stored as PNG files with signature-hashed filenames. A baseline is "approved" either by running `vizzly tdd run --set-baseline` (bulk approval) or by accepting individual screenshots from the dashboard.

**Signature** — A SHA hash derived from a screenshot's name plus its properties (browser, viewport, etc.). Used as the unique identifier for a baseline file. Example: the screenshot named `homepage` captured with `{ browser: 'chrome', viewport: '1920x1080' }` gets a different signature — and therefore a different baseline file — than the same screenshot captured with `{ browser: 'firefox', viewport: '1280x800' }`. This allows one test suite to maintain separate baselines per environment without name collisions. See [Baseline System](../concepts/baseline-system.md).

**Threshold** — The maximum acceptable CIEDE2000 Delta E difference between a baseline and a new screenshot before it's marked "failed". Default: `2.0`. Setting threshold to `0` means pixel-perfect. Setting it to `5` is tolerant of font antialiasing and minor rendering variation. Configured globally in `vizzly.config.js` or per-screenshot via the `threshold` option in `vizzlyScreenshot()`.

**Delta E (CIEDE2000)** — A perceptual color difference metric. It measures how different two colors look to a human observer, not just how different their RGB numbers are. `ΔE = 0` means identical. `ΔE = 1` is the threshold at which a trained observer can just barely detect a difference. `ΔE = 2` (Vizzly's default) corresponds to a visible but small difference. This is more meaningful than pixel-counting because a 1px antialiasing change on a dark background can have ΔE < 1 and be ignored, while a button changing from blue to green will have ΔE > 20 and always be caught.

**Honeydiff** — The diff engine used by Vizzly (the `@vizzly-testing/honeydiff` package). It doesn't just compare pixels — it finds changed pixel clusters, computes their CIEDE2000 Delta E, and filters out clusters smaller than `minClusterSize`. The output is a diff PNG highlighting changed regions and a diffPercentage representing the fraction of pixels that exceed the threshold. See [Comparison Engine](../concepts/comparison-engine.md).

**Cluster** — A contiguous group of changed pixels identified by honeydiff. Clustering prevents isolated single-pixel differences (from antialiasing or subpixel rendering) from triggering failures. The `minClusterSize` setting (default: 10) ignores any cluster with fewer than N pixels. Increase this to tolerate more rendering noise; decrease it to catch smaller changes.

**TDD Server** — The local HTTP server that Vizzly runs during test execution. It listens on port 47392 (configurable), accepts screenshots via `POST /screenshot`, finalizes runs via `POST /flush`, and serves the live dashboard. It is the hub of the local TDD workflow. See [TDD Server](../concepts/tdd-server.md).

**Daemon mode** — Running the TDD server as a persistent background process via `vizzly tdd start`. The server stays alive across multiple test runs. You run your tests separately (e.g., `npm test -- --watch`) and they connect to the already-running server. The server persists until you stop it with Ctrl-C or `vizzly tdd stop`.

**One-shot mode** — Running the TDD server for exactly one test run via `vizzly tdd run "npm test"`. The server starts, runs your test command, waits for it to complete (detected via `vizzlyFlush()`), generates a static HTML report in `.vizzly/report/`, and then exits. Useful for CI-like local runs where you want a clean report.

**Context bundle** — A structured JSON document containing visual evidence about a build or comparison. Generated by `vizzly context build <id>`. Contains screenshot metadata, diff percentages, baseline paths, comparison results, and more. Designed to be fed to LLM agents as prompt context. The `--agent` flag produces a compact summary; `--full` produces the complete payload. See [Context Bundles](../concepts/context-bundles.md).

**Build** — A collection of screenshots captured in a single test run. In local TDD mode, a "build" corresponds to one run of your test command. In cloud mode, a build is uploaded to vizzly.dev and gets a unique ID. The build ID is tracked via `VIZZLY_BUILD_ID` or auto-generated.

**Comparison** — The result of comparing one screenshot against its baseline. Each comparison has a name, status (new/passed/failed), signature, diffPercentage, and paths to the baseline/current/diff images. Comparisons are the unit of review — you accept or reject each comparison individually.

**Environment** — A label (e.g., `test`, `staging`, `production`) used to namespace baselines. Configured via `build.environment` in `vizzly.config.js` or `--environment` on the CLI. Different environments can have different baselines for the same screenshot name, allowing your staging environment to have a different approved look than production.

**Properties** — Arbitrary key-value metadata attached to a screenshot. Common properties: `browser`, `viewport`, `theme`, `locale`. Properties that are part of `signatureProperties` (configurable) are used to compute the signature hash — changing them means a different baseline is used. Properties not in `signatureProperties` are stored as metadata only.

**Flush** — The signal from the test suite that all screenshots for the current run have been submitted. Sent by calling `vizzlyFlush()` in your test teardown (e.g., `afterAll`). This triggers the server to finalize the run: write the summary, generate the report, and print results. Without a flush, the server doesn't know when tests are done.

**Auto-discovery** — The mechanism by which the client SDK finds the running TDD server without explicit configuration. When `vizzlyScreenshot()` is called and no server URL is configured, the SDK walks up the directory tree from `process.cwd()` looking for `.vizzly/server.json`. If found, it reads the port and connects automatically. This means you don't need to set any environment variables or configuration for local TDD use. See [Client SDK](../concepts/client-sdk.md).
